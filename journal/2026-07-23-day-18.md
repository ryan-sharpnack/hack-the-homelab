## What got done

1. Confirmed the VM set needed for Rule 1 testing — DC01-IRONBRIDGE (generates the events), SIEM01 (ingest/query), WS01 (requesting workstation), FW01 (gateway/DNS). DC01-HAWTHORNE and APP01 not required for this rule.

2. Built and tested LogQL for **IHA-DET-01 (Kerberoasting-indicative)**. Discovered that `event_data` isn't a set of discrete JSON fields — it's raw XML embedded as a single string value inside the parsed JSON — meaning every rule needs a `| json` stage for top-level fields (`event_id`, `computer`, etc.) plus a regex match against `event_data` itself for anything Windows nests inside `<Data Name='...'>` tags. This shaped the syntax for all five rules, not just this one.

3. Generated real traffic to verify the rule — deliberately recreated the actual root cause (an RC4-only service account via `msDS-SupportedEncryptionTypes`) rather than using offensive tooling, then requested a ticket natively via .NET's `KerberosRequestorSecurityToken` from WS01. Confirmed Event ID 4769 / `TicketEncryptionType 0x17` in Event Viewer and isolated the single matching line in Grafana Explore.

4. Converted the query into a Grafana-managed alert rule (`IHA-DET-01`), folder `IHA - Detections`, 1m evaluation interval, pending period None (appropriate for a rare/binary event rather than a threshold pattern). Confirmed the rule transitioned to Firing on retest.

5. Created a placeholder contact point (`homelab-alerts`) using Grafana's default email integration. SMTP isn't configured, so no mail actually sends — deliberately deferred to Phase 5, since real notification routing is a Phase 5 concern and today's evidence only needs the Firing state, not delivered mail.

6. Built LogQL for **IHA-DET-02 (Privileged Group Membership Change)** — Event IDs 4728/4732/4756 filtered to Domain Admins, Enterprise Admins, Schema Admins, Administrators.

7. Hit a parse error (`unexpected $end`) saving the Rule 2 alert despite the identical query validating cleanly in Explore. Traced it to the Builder UI — adjusting the `instance` label filter's operator had also silently flipped the `event_id` operator from `=~` (regex) to `=` (exact match) elsewhere in the same query, which can never match a real single-value field. Fixed by explicitly restoring `=~` on `event_id`.

8. Hit a second issue during Rule 2 verification: enabling **Security Group Management** auditing via local `auditpol` appeared to work but produced no 4728 event on test. Root cause — Default Domain Controllers Policy GPO refresh (every 5 minutes) silently reverts any Advanced Audit Policy subcategory not explicitly defined at the GPO level. Fixed by defining the subcategory directly in Default Domain Controllers Policy and forcing `gpupdate /force`.

9. Retested by adding the existing `svc-test-kerb` account to Domain Admins, confirmed 4728 in Event Viewer with correct Subject/Member fields, and confirmed the alert rule fired.

10. Started the Task 5 risk register walk — updated IHA-012 from Open to Partially Mitigated, reflecting Rules 1–2 as verified detections, with Rules 3–5 and the FIM gap noted as the remaining scope. Drafted `IHA-RB-DET-KERBEROASTING.md` and `IHA-RB-DET-PRIVGROUP-MEMBERSHIP.md`.

## Notes

Most of today's friction wasn't the detection logic itself — once the `event_data`-as-embedded-XML structure was understood, the LogQL for both rules came together quickly. The real cost was two separate infrastructure quirks that looked like detection failures but weren't: Grafana's Builder silently changing an unrelated operator during an edit, and a DC's audit policy reverting itself on the next GPO refresh cycle. Neither was hard to fix once isolated, but confirming that an alert wasn't firing because of tooling behavior — not a bad query — took real diagnostic time in both cases.

The audit policy finding is the more durable one to carry forward. Local `auditpol` changes on a domain controller don't persist unless the same subcategory is also defined at the GPO level — a gap that's invisible until you specifically test for it, and one that will resurface on Rules 3–5 for any Security log subcategory not already covered by existing policy. Also worth sitting with: updating the risk register per verified rule, rather than in one batch at the end, is producing a cleaner audit trail — each entry now has a specific date and specific evidence behind it, which is a better artifact for the case study than a single end-of-project update would have been.
