## What got done

1. **Resolved the `qemu:///session` fallback permanently.** `groups $USER` confirmed `libvirt` membership was already in place. Setting `LIBVIRT_DEFAULT_URI` initially failed to take effect after `source ~/.bashrc` — the shell in use turned out to be zsh, not bash, evidenced by bash-completion errors and a garbled prompt when the wrong file was sourced. Corrected by adding the export to `~/.zshrc` instead. A subsequent typo (`LIBVIRT_DEFFAULT_URI`) caused a second silent failure before being caught via `grep` and corrected. Verified via `virsh list --all` running cleanly with no `-c qemu:///system` flag needed.

2. **Investigated SIEM01 outbound connectivity ahead of the OpenSSH task.** Initial concern was whether an isolated single-NIC VM would need a second NAT interface added to reach the internet for package installs. Determined this would violate the segmentation model (all traffic should route through FW01) and was unnecessary — the real issue lay elsewhere.

3. **Found and removed a stale, undocumented address** (`192.168.123.10/24`) on host bridge `virbr1`, discovered while investigating why `10.50.10.1` answered pings with FW01 powered off. Confirmed inert via NAT table review and a config-file search across netplan, systemd-networkd, and NetworkManager sources before removing it — no rationale existed for it in any design document, likely a leftover from early host provisioning.

4. **Traced and corrected the actual root cause: SIEM01's default gateway was misconfigured at OS install time.** Ubuntu's Subiquity installer had set the gateway to `10.50.10.1` instead of the documented `10.50.10.254` (FW01's LAN interface). This went unnoticed because libvirt's isolated `vlan10-corp` network independently assigns `10.50.10.1` to the host bridge, so SIEM01 appeared to have a working gateway even with FW01 shut off — a misconfiguration masked as a false success rather than surfaced as a clean failure. Corrected via netplan, with cloud-init network management disabled (`network: {config: disabled}`) so the fix survives reboot. Verified via an actual reboot test, not just a live session check.

5. **Verified DC01-IRONBRIDGE and DC01-HAWTHORNE do not share this problem.** `ipconfig /all` on both confirmed correct gateways (`10.50.10.254` and `10.50.30.254` respectively) — Windows Server's install process doesn't share Subiquity's default-gateway quirk, so no fix was needed, but worth ruling out rather than assuming.

6. **Added three scoped pfSense egress rules on the LAN tab** for SIEM01 (`10.50.10.20` → Any/WAN: TCP 80, TCP 443, UDP 53) so it can reach package repositories for updates, consistent with the environment's least-privilege, default-deny posture. ICMP was deliberately left unpermitted — it was never an actual requirement, only a diagnostic convenience. Verified via a successful `curl` to the Ubuntu archive mirror.

7. **Completed the OpenSSH install on SIEM01** (`openssh-server`, enabled and confirmed listening on port 22) and added a corresponding pfSense LAN rule. Discovered via `ip route get` that SSH traffic from the management host to SIEM01 is switched at Layer 2 across the shared `virbr1` bridge and never actually traverses FW01 — the pfSense rule is correct and worth keeping for genuinely external clients, but isn't the enforcement mechanism for this particular path. Test connection succeeded.

8. **Logged three new risk register entries** (IHA-008 through IHA-010) covering the stale bridge address, the gateway misconfiguration, and the egress rule gap — each with a distinct root cause, tracked start to finish (identified → verified/mitigated). Appended a note to IHA-006 documenting that hypervisor-to-guest access bypassing FW01 is expected, inherent behavior, not a new gap.

9. **Wrote two new runbooks:** `IHA-RB-LNX-STATIC-NETWORK-CONFIG.md` (gateway verification and netplan/cloud-init correction procedure, reusable for WS01/APP01 in Phase 3) and `IHA-RB-LNX-SSH-ENABLEMENT.md` (SSH install/verify/firewall-rule steps, including the hypervisor-vs-external-client nuance from step 7).

10. **Updated `network-addressing-and-firewall-rules.md`** with the three new egress rules and a permanent clarifying note distinguishing `.1` (host bridge, libvirt-assigned) from `.254` (the only valid VM gateway) — intended to prevent this exact confusion from recurring.

## Notes

Wazuh install has now slipped a second session in a row — this time displaced by a single planned task (adding OpenSSH to SIEM01) that unfolded into two real, previously undocumented misconfigurations: a stale bridge address and a gateway set wrong at install time, the second of which was actively masking itself as working. Neither was hypothetical; both would have caused real problems down the line if left unfound; and this is real practice at exactly the kind of quiet, easy-to-miss gap that matters in production healthcare networks.

That said, worth naming plainly: the pattern of "one small task uncovers a deeper issue, and the deeper issue eats the whole session" has now happened twice back to back. The diagnostic habit is genuinely valuable and worth keeping. What's worth adding alongside it is a rougher time-box on unplanned detours — investigate enough to root-cause and log it properly, but consciously decide when to stop digging and carry the remainder forward, rather than letting it consume the day's original goal by default. Phase 2's actual exit criteria (SIEM ingesting from both DCs) hasn't moved in two sessions; that's the thing to protect next time.
