# Day 7 — Phase 2: Foundation Build

**Date:** July 12, 2026

## Goals for today

- Rebuild the IRONBRIDGE.LOCAL ↔ HAWTHORNE.LOCAL one-way forest trust from a clean state, after last night's session ran into errors that burned through free-tier usage with Claude AI before they could be resolved
- Confirm conditional forwarders resolve correctly in both directions before touching the trust
- Verify the resulting trust matches the Phase 1 design spec — Forest-wide authentication, not Selective Authentication, since that scoping-down is a planned Phase 4/5 redesign deliverable, not a Phase 2 requirement

## What got done

1. **Cleared a stale DNS forwarder.** Recreating the HAWTHORNE.LOCAL conditional forwarder on DC01-IRONBRIDGE initially failed with a `ResourceExists` error — a leftover zone from last night's troubleshooting. Removed the stale zone on both DCs (`Remove-DnsServerZone -Force`) and recreated the forwarders cleanly.
2. **Verified DNS resolution in both directions.** `Resolve-DnsName` confirmed DC01-IRONBRIDGE can resolve `hawthorne.local` via 10.50.30.10, and DC01-HAWTHORNE can resolve `ironbridge.local` via 10.50.10.10 — the prerequisite for Kerberos/SRV lookups across the trust.
3. **Found a leftover, mismatched trust object.** `Get-ADTrust` on DC01-IRONBRIDGE showed an existing HAWTHORNE.LOCAL trust with `SelectiveAuthentication: True` — a remnant from last night that didn't match today's Phase 2 spec (Forest-wide authentication).
4. **Removed the trust from both sides.** `netdom /remove` initially failed with a credential error until the other domain's admin credentials were supplied explicitly (`/Ud:` / `/Pd:*`), which turned out to remove both sides' objects in a single command.
5. **Rebuilt the trust via the New Trust Wizard**, run separately on each DC ("This domain only"): One-way outgoing + Forest-wide authentication on DC01-IRONBRIDGE, One-way incoming on DC01-HAWTHORNE, using a matching trust password on both sides.
6. **Verified the result.** GUI Validate succeeded on both domains; `Get-ADTrust` confirmed `Direction: Outbound`, `ForestTransitive: True`, `SelectiveAuthentication: False` — matching the Phase 2 spec exactly. Confirmed name suffix routing enabled by default (no action needed).
7. **Documented the procedure.** Wrote `runbooks/IHA-RB-WIN-CROSSFOREST-TRUST.md`, capturing the full rebuild procedure and a troubleshooting section for the three leftover-state issues hit today, so a future rebuild (or the Phase 4/5 redesign) doesn't require rediscovering them.
8. Resolved IHA-007. Re-verified via virsh -c qemu:///system net-dumpxml on all three libvirt networks (vlan10-corp, vlan20-hosp, vlan30-hawth) — none had an active <dhcp> block, despite the earlier ps aux | grep dnsmasq check showing all three processes running. Determined those processes were DNS-only, not DHCP-serving; the original check couldn't distinguish the two. No libvirt config changes required. Updated the risk register, preserving the original 2026-07-08 note and appending the corrected 2026-07-12 finding rather than overwriting.
9. Created SIEM01 VM. 4GB RAM / 2 vCPU / 60GB disk, virtio NIC on vlan10-corp. Installed Ubuntu Server 22.04.5 — chosen over Ubuntu 26.04 and Rocky 10.2 for the most mature Wazuh 4.14 compatibility. Hostname siem01, static IP 10.50.10.20/24, gateway 10.50.10.1, DNS 1.1.1.1. OpenSSH server install skipped for tonight, carried to next session. Resolved a cosmetic "failed unmounting /cdrom" installer error by disconnecting the ISO via virt-manager; booted and logged in cleanly.
10. Snapshotted SIEM01 at siem01-pre-wazuh-install — clean baseline before Wazuh install begins.

Session evidence saved to `evidence/2026-07-12/`.

## Notes

Today's three snags — a stale forwarder, a leftover trust with the wrong auth scope, and a cross-forest credential requirement on removal — were all leftover state from last night's session, not new problems with the underlying design. Checking actual state first (`Get-ADTrust`, `Get-DnsServerZone`) before assuming a fix was needed, rather than re-guessing at commands, is what actually got today's rebuild done cleanly. That diagnostic habit is worth more long-term than getting the config right on the first attempt.
