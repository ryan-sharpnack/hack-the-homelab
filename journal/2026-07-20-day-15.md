# Day 15 — Phase 3: Clinical Environment Build Kickoff (WS01)

**Date:** July 20, 2026

## Goals for today

- Deploy WS01 (Windows 11) per Phase 3 scope
- Confirm host prerequisites (`ovmf`, `swtpm`, `swtpm-tools`) and RAM headroom before provisioning
- Decide APP01's architecture ahead of its build
- Get WS01's OPT1 firewall rules in place, and APP01's if time allows

## What got done

1. Confirmed WS01's placement — VLAN20-Hosp, `10.50.20.10`, gateway `10.50.20.254` — against the network architecture doc, and verified the libvirt network naming (`vlan20-hosp`) via `virsh net-list --all` before attaching the NIC, rather than assuming it matched the doc's convention.

2. Reviewed the OPT1 rule set ahead of provisioning and found the local firewall doc copy was stale — still showing the legacy Wazuh 1514/1515 rules as active rather than disabled. Pulled the current GitHub version, which correctly reflected the Wazuh→Loki pivot (legacy ports disabled, `3100` Loki push rule added on OPT2), and reconciled the local copy to match.

3. Created the WS01 VM in virt-manager: Windows 11, UEFI x86_64 firmware, emulated TPM 2.0 (model CRB). First install attempt failed — `unsupported configuration: only a single TPM non-proxy device is supported` — caused by virt-manager auto-adding a default TPM alongside the one added manually. Removed the duplicate and proceeded. Sized memory at 4096MB, revised down from an initial 6144MB estimate once RAM headroom was checked against DC01-IRONBRIDGE running concurrently and APP01's build still ahead.

4. Windows 11 installation was interrupted by a forced power-off mid-install; resumed cleanly on reboot. Skipped the network driver install screen — confirmed after the fact this was fine, since the emulated Intel 82574L NIC has a driver built into Windows already (no virtio driver injection needed, unlike what would be required for a virtio NIC).

5. Hit Windows Server/11 evaluation license expiration on both DC01-IRONBRIDGE and WS01 during this session.

   **Windows licensing status — DC01-IRONBRIDGE and WS01.** Both hosts run Windows Server/11 evaluation media. DC01-IRONBRIDGE has used its final rearm as of today — no further resets available; current grace period is the last one before full expiration and automatic hourly shutdown behavior begins. WS01 has one rearm remaining. This is a lab-continuity constraint, not an environment security finding, so it's tracked here rather than in the risk register. Action: plan to reinstall DC01-IRONBRIDGE with fresh eval media (or apply a licensed key via DISM) before its current grace period lapses, and revisit WS01's status before its own rearm budget runs out.

6. Decided APP01's architecture ahead of its build: a lightweight, non-domain-joined Rocky Linux VM authenticating via LDAPS bind against IRONBRIDGE.LOCAL through a dedicated `svc-app01` service account in `OU=ServiceAccounts`, rather than a full Windows domain member. Chosen over full domain join for a lighter RAM footprint and a smaller firewall rule set — and because it keeps Phase 5's Kerberoasting attack surface on the AD account itself (via a registered SPN) rather than requiring a third Windows host to carry it.

7. Wrote and applied the OPT1 rule set in the pfSense GUI: 10 rules for WS01 (DNS, Kerberos, RPC Endpoint Mapper, LDAP, SMB, Kerberos password change, LDAPS, Global Catalog LDAP/LDAPS, RPC dynamic range — all `10.50.20.10 → 10.50.10.10`) and 2 for APP01 (DNS + LDAPS only, `10.50.20.20 → 10.50.10.10`, reflecting the lighter architecture decided above). Verified against spec in the GUI with no discrepancies.

## Notes

Most of today was infrastructure mechanics rather than anything conceptually new — a duplicate TPM device, RAM budgeting across four concurrent VMs, two separate eval-license grace periods running out in the same session. None of it was hard to fix, but all of it cost real time, and DC01-IRONBRIDGE's rearm situation is worth sitting with: there is no more slack left on that host's license. That's not a crisis, but it's a real constraint on the rest of this project now, tracked deliberately outside the risk register since it's a lab-continuity issue, not a finding about IRONBRIDGE.LOCAL's actual security posture.

The one design decision that mattered today was APP01's architecture. Defaulting to a full domain join would have been the easy, unexamined choice; going with a lighter LDAPS-bind client instead is a better story precisely because it's visible in the firewall rule count — 2 rules instead of 10 — which is a concrete, defensible artifact of a deliberate choice rather than something that has to be explained in prose alone.
