# Day 6 — DC01-HAWTHORNE Build Kickoff

**Picking up from Day 4**

Started on the first carried-over item from Day 4's open list: building DC01-HAWTHORNE, which will be promoted as the forest root for HAWTHORNE.LOCAL. VM created to the same spec as DC01-IRONBRIDGE (2 vCPU / 4096MB / 60GB, Q35 chipset), with one deliberate change this time — UEFI firmware instead of BIOS, moving the lab environment closer to a production-representative baseline.

**UEFI boot hiccup**

Downloaded the Windows Server 2019 evaluation ISO from Microsoft and attached it, but the VM didn't auto-launch Setup on first boot — it dropped into the UEFI Interactive Shell instead. This is expected behavior for a fresh UEFI VM with no boot entry yet, not a fault in the ISO or VM config. Diagnosed and resolved by mapping the CDROM (`fs0:`), confirming the `EFI` directory was present, and manually invoking the bootloader.

One wrinkle worth flagging for future builds: my first attempt used forward slashes (`/EFI/boot/bootx64.efi`), which the UEFI Shell didn't recognize as a path at all — it read it as an unknown command rather than throwing a path error, which cost a few minutes before I caught that the Shell expects backslash syntax like Windows (`\EFI\boot\bootx64.efi`). Once corrected, Setup launched normally.

Adding this as a new entry in **IHA-RB-WIN-DC-FOREST-PROMOTION.md**, Known Non-Issues section, alongside the DNS delegation warning and Event 4014 items from the IRONBRIDGE build. Also setting CDROM ahead of disk in the VM's boot order so this doesn't recur on reboot.

**OS install, base config, and forest promotion**

Completed the remainder of the build after the boot fix: Custom install (clean disk, no prior OS, so Upgrade wasn't a valid option), logged in as Administrator, renamed the computer to `DC01-HAWTHORNE`, and set the static IP (10.50.30.10/24, gateway 10.50.30.254, DNS 127.0.0.1) — confirmed via `ipconfig /all`. Installed AD DS via Server Manager, then promoted the server as the root of a new forest: `HAWTHORNE.LOCAL`, NetBIOS name `HAWTHORNE` (auto-derived, no override needed), Global Catalog and DNS Server enabled, DNS delegation warning accepted (expected for a standalone root), default paths.

**Post-promotion verification**

Ran the full verification pass from the runbook:

- `Get-ADDomain` — DomainMode `Windows2016Domain`, PDCEmulator/RIDMaster/InfrastructureMaster all correctly pointing to `DC01-HAWTHORNE.HAWTHORNE.LOCAL`
- `Get-DnsServerZone` — both `HAWTHORNE.LOCAL` and `_msdcs.HAWTHORNE.LOCAL` present and AD-integrated
- `net share` — SYSVOL and NETLOGON both present, correct share paths
- `nslookup hawthorne.local` — resolved cleanly to 10.50.30.10
- Event Viewer — one new transient warning not seen on IRONBRIDGE: Event 4013 (DNS-Server-Service), logged while DNS Server waits for AD DS to signal initial sync completion. Self-clears once sync finishes — a different flavor of "expected noise" than IRONBRIDGE's Event 4014 (that one was a boot-order race; this one is a sync-timing wait).

DC01-HAWTHORNE is fully built, promoted, and verified.

**Runbook updated**

Folded today's findings into **IHA-RB-WIN-DC-FOREST-PROMOTION.md**:

- Added a preventive step to VM Creation — set boot order to CDROM-first before first power-on, so future builds skip the UEFI Shell fallback entirely
- Added the UEFI Shell fallback itself to Known Non-Issues, with the corrected backslash path syntax, and a note that it's a one-time first-boot issue that doesn't recur once Windows is installed
- Added Event 4013 to Known Non-Issues, distinguished from Event 4014

Kept this as an addition to the existing runbook rather than a new file — HAWTHORNE followed the identical DC-build-and-promote procedure as IRONBRIDGE, so the runbook gets more battle-tested rather than fragmented across per-machine files.
