## What got done

1. Installed Grafana Alloy on WS01 via the same isolated-VM ISO transfer method used on both DCs in Phase 2 — `virt-manager` CD-ROM attach rather than the DCs' original `genisoimage`/`virsh` CLI approach, since WS01 has no outbound internet either (confirmed via a failed DNS lookup in Edge before starting). Attached to `sdc`, since `sda` and `sdb` were already occupied by the OS disk and a leftover Windows install ISO.

2. Drafted an Alloy config from scratch initially, then caught that it didn't match the DCs' actual syntax — corrected by locating saved screenshots of the original DC install runbook rather than trusting recall of the file contents. Rebuilt using `loki.source.windowsevent` blocks for Security/System/Application, `use_incoming_timestamp = true`, and `instance = constants.hostname`, matching both DCs exactly (`job=windows_eventlog`, underscore, not hyphen).

3. Hit a permissions error editing `config.alloy` — Notepad wasn't launched elevated before opening the file. Resolved by launching Notepad directly from an elevated PowerShell session instead. Ingestion confirmed in Grafana Explore (`{instance="WS01"}`, 92 lines in the first hour). Added a dedicated "Windows Event Log - WS01" panel to the IHA Phase 2 dashboard, kept separate from "Both DCs" rather than merged, for clarity ahead of APP01 joining later. Confirmed both panels showing correct, non-overlapping host data after a page refresh.

4. First `auditpol` check on Kerberos audit policy showed both subcategories (Authentication Service, Service Ticket Operations) as Success-only, not Success and Failure as intended. Investigated and found no dedicated GPO existed at all — Default Domain Controllers Policy showed all four Account Logon subcategories as "Not Configured." The Success-only baseline was a local Windows default, not GPO-driven.

   **GPO created:** `IHA-Kerberos-Audit-Policy` — both subcategories set to Success and Failure, "Force audit policy subcategory settings" enabled (Local Policies → Security Options) to prevent legacy category settings from silently overriding the advanced config, linked to the Domain Controllers OU. Re-verified via `auditpol /get /category:"Account Logon"` — both subcategories confirmed Success and Failure.

5. Built FIM per the plan decided 2026-07-15 (IHA-012) — Windows Object Access auditing feeding the existing Loki pipeline, no new tool. Required separate GPOs per forest (`IHA-FIM-Object-Access-Audit-Policy`) since GPOs don't cross the HAWTHORNE→IRONBRIDGE trust — Audit File System enabled for Success and Failure on both, linked to each forest's own Domain Controllers OU.

6. Applied the actual SACL to `SYSVOL\domain\Policies` on both DCs. The audit category alone logs nothing without an object explicitly flagged — used advanced permissions (Create files/write data, Create folders/append data, Delete subfolders and files, Delete, Change permissions) scoped to Everyone, since Delete and Change Permissions don't appear in the basic-permissions list at all.

7. Verified on IRONBRIDGE first: created and deleted a throwaway GPO (`IHA-SACL-Test-Delete-Me`), confirmed 8 lines of event ID 4663 landing in Loki within the minute, tied directly to the test GPO's GUID folder path with `AccessMask: DELETE` visible. HAWTHORNE needed a second pass — deleted the test GPO from the wrong location on the first attempt, so nothing landed on the first check despite the GPO and SACL both being correctly configured (`auditpol` already confirmed Success and Failure beforehand). Corrected and re-ran; 4663 confirmed landing for `instance=DC01-HAWTHORNE`. FIM gap closed and verified on both forests.

8. Updated the risk register: **IHA-004** kept Open (audit visibility isn't the same as mitigation — that's the still-pending Kerberoasting LogQL rule) with a verified note on the GPO build. **IHA-012** moved Open → Mitigated, Date Resolved 2026-07-22 — both stated exit conditions met (SACL configured, 4663 verified on both hosts). Split the note to separate FIM (closed today) from alerting (still open, deliberately not rolled into this closure).

9. **Decided to deprioritize the Compliance/GRC Mapping workbook going forward.** Adds documentation overhead without proportional return for a non-GRC-track analyst role — the Risk Register alone already carries the HIPAA-relevant risk-based-thinking signal a healthcare-focused hiring manager would look for. Will keep the Risk Register current; won't expand the GRC Mapping further.

10. **Decided to defer the alerting layer (Grafana notification wiring) to Phase 5**, rather than building it alongside the 5 LogQL detection rules in Task 4. The detection queries stay on schedule for tomorrow. Wiring them to actual notification firing lands better demonstrated live against a real Phase 5 simulated attack than configured and tested in isolation — same work, stronger case-study narrative.

11. Took two snapshots: `dc01-hawthorne-phase4-task3-fim-verified` (offline/external, post graceful shutdown, capturing GPO + SACL + verified FIM state) and `dc01-ironbridge-phase4-task2-task3-verified` (live, ahead of tomorrow's Task 4 work, capturing the Kerberos audit GPO and FIM SACL state).
