## What Actually Got Done

**Addressing verification turned into a full host-network investigation.** What started as "confirm .254 on each interface" surfaced a real finding: the KVM host's libvirt bridges (virbr1/2/3) hold `.1` addresses on each VLAN subnet, which initially looked like it could indicate NAT/routing at the host level, bypassing FW01. Traced this down through several dead ends (wrong `virsh` connection scope, a stray Docker network from a prior project) before confirming via `iptables -L LIBVIRT_FWI/FWO` that host-level forwarding between VLANs is unconditionally rejected — `ip_forward = 1` at the kernel level, but neutralized by libvirt's own firewall policy. FW01 remains the sole inter-VLAN path, as designed. pfSense console confirmed LAN/OPT1/OPT2 correctly holding `.254` on all three subnets.

**Built the full pfSense firewall rule set** — 24 explicit rules across LAN, OPT1, and OPT2, on a confirmed default-deny base (removed the factory-default "allow LAN to any" rules first). Covers the IRONBRIDGE↔HAWTHORNE AD trust (DNS, Kerberos, LDAP, SMB, RPC, and the dynamic RPC range) in both directions, plus SIEM01 log ingestion (Wazuh ports 1514/1515) from both VLAN20 and VLAN30. Caught and fixed a transposed-digit typo in the RPC dynamic range (41952 → 49152) before it shipped.

**Documented the WAN anti-spoofing filter decision** — disabled "Block RFC1918" and "Block bogon networks" on FW01's WAN interface, since the nested-lab WAN uplink is itself RFC1918 space via libvirt's NAT network. Written up as a permanent, accepted design decision rather than a workaround.

**Risk Register:** populated Likelihood/Impact for IHA-001 through IHA-005 with documented rationale for each rating. Upgraded IHA-006's evidence standard (iptables ruleset inspection vs. config review). Opened IHA-007 — libvirt's dnsmasq is actively serving DHCP on all three VLANs, which will collide with the Windows DHCP server role once the DCs are built in the next session.

**Compliance/GRC Mapping:** moved GRC-001 (network segmentation) from Planned to In Progress, with the firewall/VLAN evidence attached. Held back from marking it Implemented since there's no live traffic through these rules yet — no DCs, no SIEM — so the control is real but not yet end-to-end validated.
