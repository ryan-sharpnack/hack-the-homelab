# Day 11 — Phase 2: Retention Implementation & Formal Close

**Date:** July 16, 2026

## Goals for today

- Implement the Loki retention policy decided but not yet executed on Day 10 (30-day/720h active retention)
- Update the `siem01-phase2-exit-baseline` snapshot description to reflect the change
- Confirm the two remaining Day 10 carryover items are actually closed, and formally close Phase 2

## What got done

1. Attempted to pull Loki's running config via `docker exec iha-loki-loki-1 cat /etc/loki/local-config.yaml`. First failed on a container-name typo (a digit `1`, not the letter `l`, easy to misread in a terminal font). Corrected, then failed again with `exec: "cat": executable file not found in $PATH`.

2. Root cause: Grafana's official Loki image ships without a shell or BusyBox utilities by design — documented upstream behavior, not a misconfiguration on this end. `docker exec` fundamentally can't run anything inside this image. The failed command's shell redirect (`>`) had already created `loki-config.yaml` regardless of the command's failure — the file existed, but contained the error text itself, not real config. Caught before pasting the retention block into it, which would have produced invalid YAML on top of garbage content.

3. Re-extracted the actual config using `docker cp iha-loki-loki-1:/etc/loki/local-config.yaml ./loki-config.yaml` instead — copies through Docker's filesystem layer directly, sidestepping the missing-shell problem entirely. Confirmed the real default config this time before proceeding.

4. Checked for existing `limits_config`/`compactor` blocks before adding new ones — none present in the true default, so no merge conflict this time (unlike the duplicate-key issue hit in `docker-compose.yml` on Day 10). Added:
   - `limits_config.retention_period: 720h` — 30-day active retention. Documented explicitly as a lab disk-space decision, not a represented production recommendation; a real healthcare environment would likely retain longer.
   - `compactor` block: `working_directory: /loki/compactor`, `retention_enabled: true`, `delete_request_store: filesystem`.

5. Validated the edited YAML with `python3 -c "import yaml; yaml.safe_load(open('loki-config.yaml'))"` before saving — cheaper to catch a syntax error there than after a failed container start.

6. Mounted the file read-only in `docker-compose.yml` (`./loki-config.yaml:/etc/loki/local-config.yaml:ro`), ran `docker compose up -d`, confirmed via `docker logs` that Loki started clean — no schema or compactor errors.

7. Updated the `siem01-phase2-exit-baseline` snapshot description via `virsh snapshot-edit` (metadata-only — does not retake the snapshot or alter the captured disk state). Appended rather than rewrote, and explicitly noted that the originally captured state predates this change, so restoring that snapshot would require reapplying the retention config and compose mount by hand.

8. Confirmed both remaining Day 10 carryover items are closed: Doc 08 marked Phase 2 complete, and today's evidence filed to its dated folder.

9. With retention implemented, the snapshot description accurate, and both loose ends closed — **Phase 2 is formally closed as of today**, within the "today, tomorrow at the absolute latest" window set on Day 10.

## Notes

The `docker exec` failure today was a small, low-stakes version of the same lesson underlying all of Phase 2's pivot: don't assume a tool behaves the way the last tool did. Wazuh's image had a full shell; Loki's deliberately doesn't. Neither is wrong — but reaching for `docker exec` out of habit, rather than checking what this specific image actually supports, is exactly the kind of assumption that costs a cycle. Same shape of mistake as the original gateway `.1` vs `.254` masking issue back on Day 8, just smaller and faster to catch.

Phase 2 in full ran longer than originally scoped — a SIEM install that slipped two sessions, then failed outright, then required a genuine architecture pivot under a self-imposed deadline. The honest deliverable of this phase isn't just "SIEM ingesting from both DCs." It's a full, real example of keeping infrastructure and documentation in agreement under time pressure, with a complete audit trail of what changed and why — which is closer to what the job this project is aimed at actually looks like than a phase that went according to plan the first time would have been.
