# Day 1 — Phase 2: Foundation Build
**Date:** July 6, 2026

## Goals for today
- Clear out all VMs and volumes from the prior build to start Phase 2 clean
- Stand up isolated VLAN networks for Corporate (10), Hospital (20), and Hawthorne (30)
- Install and configure FW01 (pfSense CE) as the routing/policy boundary between all three VLANs
- Get FW01's interfaces assigned and addressed correctly before building any downstream VMs

## What got done
1. **Clean slate.** Deleted all pre-existing VMs and their associated `.qcow2` disk volumes from the previous build. Confirmed via Storage → refresh that no orphaned volumes remained.
2. **Virtual networking.** Created three isolated libvirt networks in Virtual Machine Manager to represent the VLAN segments:
   - `vlan10-corp` — 10.50.10.0/24 (Corporate)
   - `vlan20-hosp` — 10.50.20.0/24 (Hospital clinical)
   - `vlan30-hawth` — 10.50.30.0/24 (Hawthorne legacy)

   All three configured as **Isolated** mode with DHCP disabled — no libvirt-managed routing or address assignment, so FW01 is the only path between segments and out to the internet. Kept the existing `default` NAT network as FW01's WAN leg only.
3. **FW01 build.** Installed pfSense CE 2.7.2-RELEASE (amd64) with UFS partitioning (chosen over ZFS to keep memory overhead low on a 1GB-RAM firewall VM). Attached four NICs at creation time: `default` (WAN), `vlan10-corp` (LAN), `vlan20-hosp` (OPT1), `vlan30-hawth` (OPT2).
4. **Interface assignment.** Ran the console assignment wizard — declined VLAN tagging (segmentation is handled by separate isolated networks, not 802.1Q) — mapped em0–em3 to WAN/LAN/OPT1/OPT2 in order.
5. **Found and fixed an IP conflict.** Initially set FW01's internal interfaces to the `.1` address on each VLAN, matching the diagram convention. Discovered that libvirt itself already assigns `.1` on each isolated network's bridge interface (`virbr1`/`virbr2`/`virbr3`) even with DHCP disabled — meaning FW01 and the host bridge would collide on the same address. Confirmed via `ip addr show virbr1` on the Kali host. Re-addressed FW01's LAN/OPT1/OPT2 interfaces to `.254` on each subnet instead, to avoid the host-owned address.
6. **Kept webConfigurator on HTTPS.** Declined the prompt to revert to HTTP — self-signed cert warning is expected and accepted; plaintext admin access isn't a defensible choice even in an isolated lab, given the project's whole premise.

## Also today
Investigated a host-level architectural question that came up while verifying FW01: since the KVM host sits on all three VLAN bridges plus WAN, could it route between VLANs directly, bypassing FW01? Verified via `iptables` that libvirt's default isolated-network rules already block this (explicit REJECT-all on each VLAN bridge in the relevant chains). Logged as **IHA-006** — a "verified, not exploitable" finding rather than a bug fix. Full technical detail and evidence screenshots are in `runbooks/IHA-RB-NET-Foundation.md`; formal risk register entry (Excel) to follow tomorrow.

Session evidence saved to `evidence/2026-07-06/`.
