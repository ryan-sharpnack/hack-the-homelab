# Day 16 — Phase 3 Close & Phase 4 Groundwork: WS01 Domain Join, APP01 Build, LDAPS Diagnosis

**Date:** July 21, 2026

## What got done

1. Diagnosed WS01's initial `nslookup` timeout to DC01-IRONBRIDGE as FW01 being powered off, not a DNS or rule problem — VLAN10 and VLAN20 are only reachable through FW01, unlike same-subnet gateway pings, which the host bridge can answer on its own. Powered FW01 on; retest succeeded (`Test-NetConnection` on 389 and 445, clean `nslookup`).

2. Confirmed the OU structure (`OU=Corporate`, `Hospital`, `ServiceAccounts`, `Admins`) had never actually been created, despite being part of the Phase 1 identity design — built all four via `New-ADOrganizationalUnit`. Created seed user `asmith` in `OU=Hospital` as WS01's day-to-day login identity.

3. Domain-joined WS01 to IRONBRIDGE.LOCAL using domain admin credentials for the join itself; verified with `whoami`, `Get-ComputerInfo`, and the definitive check, `nltest /sc_query:IRONBRIDGE.LOCAL`, confirming a healthy secure channel (`NERR_Success`). Logged in as `asmith` going forward — not Administrator, not the local account.

4. Snapshotted WS01 (`ws01-domain-joined-baseline`) as a clean checkpoint before Alloy and GPO hardening land on top of it.

5. Built APP01: Rocky Linux 10.2 minimal, VirtIO disk, BIOS firmware (no TPM/UEFI requirement, unlike WS01). Static IP `10.50.20.20/24`, gateway `10.50.20.254`, DNS `10.50.10.10` set during install. Verified post-install via `ip addr`, `ip route`, `/etc/resolv.conf`, and `hostnamectl`.

6. Created `svc-app01` in `OU=ServiceAccounts` and registered an SPN (`HTTP/app01.ironbridge.local`) via `setspn` — the piece that actually makes it Kerberoastable in Phase 5, independent of whether LDAPS itself is functional.

7. Attempted to verify APP01's LDAPS bind end to end and surfaced a real infrastructure gap in the process. `openssl s_client -connect 10.50.10.10:636` showed the TCP handshake completing (`CONNECTED`) but the TLS session resetting instantly (`errno=104`, `Cipher is (NONE)`) — port open, nothing behind it. `Get-WinEvent` on the DC confirmed via Event 1220 that AD "was unable to obtain a certificate," error `8009030e` (`SEC_E_NO_CREDENTIALS`). Root cause: `New-SelfSignedCertificate` stores its key via CNG, which `lsass`/Schannel can't access for LDAPS binding — it needs the legacy CSP instead. Rebuilt the certificate via `certreq` with an INF forcing `ProviderName = "Microsoft RSA SChannel Cryptographic Provider"`, per Microsoft's documented method; retested post-reboot, and the same failure signature persisted. Made a deliberate call to stop chasing it further today — Kerberoasting, the thing Phase 5 actually needs from `svc-app01`, runs over port 88 and doesn't depend on LDAPS at all. Logged as **IHA-013**, deprioritized; full AD CS deployment flagged as the longer-term remediation.

8. **APP01 outbound package install — temporary exception, reverted.** APP01's original firewall footprint (2 OPT1 rules: DNS + LDAPS to DC01-IRONBRIDGE only) had no path to public repositories, so `dnf install openldap-clients` failed on both DNS resolution and egress. Added 3 temporary OPT1 rules (TCP 80/443, UDP 53, scoped to `10.50.20.20` → Any/WAN) plus a secondary public DNS entry (`1.1.1.1`) alongside the existing AD DNS server, installed the package, then disabled the 3 rules (not deleted, same pattern as the legacy Wazuh rules) and removed the public DNS entry. APP01 is back to its minimal 2-rule footprint. Repeated the identical reopen/install/revert cycle a second time for `openssl`, needed to test the LDAPS certificate directly above. Reference for next time: any future `dnf` work on APP01 will need this same temporary reopening — re-enable the 3 disabled OPT1 rules and re-add `1.1.1.1` to `/etc/resolv.conf` (via `nmcli con mod enp1s0 +ipv4.dns "1.1.1.1"`), do the install, then revert both.

9. Audited the full OPT1/OPT2 rule set against intended design. Everything checked out except one thing caught mid-review: the 3 temporary APP01 egress rules from step 8's second pass had been left enabled after the `openssl` install rather than disabled. Corrected before moving on.

10. Reinstalled and reactivated Nessus on the Kali host (prior license had lapsed). Ran a Basic Network Scan baseline against WS01 and APP01 while VLAN20 is still intentionally flat — this is the "before" evidence for the Phase 4/5 redesign narrative. Results: WS01 came back clean (3 info-level findings only). APP01 flagged OpenSSH 9.9 as High (CVSS v3.0 8.1; CVE-2026-35385/-35386/-35387/-35388/-35414; no known public exploits). Logged as **IHA-014**, deferred as a discrete patch-and-rescan cycle rather than blocking today's close-out, given no active exploitation path exists.

11. Snapshotted APP01 (`app01-ldaps-svcaccount-baseline`) capturing full state: networked, `svc-app01` + SPN in place, IHA-013 and IHA-014 both logged as known open items at this checkpoint.

12. Updated the Risk Register: added IHA-013 and IHA-014 (both Open), matching existing formatting exactly. Progress Summary now reflects 14 total risks / 8 Open / 4 Mitigated / 2 Verified-Not-Exploitable, computed live via formula rather than hardcoded. Mirrored both entries into the GitHub Doc 06 (Threat Model & Risk Register), and added **T-8 (Unpatched exposed service)** to the Threat Model table — IHA-014 represents a generalizable attacker technique the way IHA-006 and IHA-012 did, unlike IHA-013, which stays Risk-Register-only as an implementation-specific gap. Also corrected a status error on IHA-012 (confirmed Open, not Mitigated — it's the still-unresolved consequence of the Mitigated IHA-011 pivot) and updated its stale "Phase 3" cross-reference to "Phase 4."

13. Found that Doc 07 (Network Addressing & Firewall Rule Set) hadn't been updated for any of today's or yesterday's OPT1 changes — still showing only the two disabled legacy Wazuh rules. Drafted the full replacement OPT1 table (WS01's 11 active rules, APP01's 2 active plus 3 disabled temporary egress rules) and a new Design Notes paragraph documenting the reopen/revert egress pattern from step 8, so it's captured in the doc and not just the journal.

## Notes

Today's most useful material wasn't any single fix — it was the LDAPS troubleshooting arc as a whole. `errno=104` with `Cipher is (NONE)` after a completed TCP handshake is a specific, recognizable signature, and chasing it from that symptom through Event 1220 to the actual CNG-vs-CSP root cause is a real diagnostic methodology, not a one-off fix. That's worth its own runbook rather than being buried in this entry — the reasoning chain is more valuable documented on its own than folded into a day's prose.

The other thing worth naming honestly: today included two separate documentation-drift catches — Doc 07 not reflecting a week and a half of OPT1 changes, and IHA-012's status being wrong on GitHub — that had nothing to do with today's actual build work. Both are exactly the kind of small thing default-deny discipline doesn't catch on its own; only reviewing against the doc does. Worth treating that review pass as a standing habit going into Phase 4, not a one-time catch-up.
