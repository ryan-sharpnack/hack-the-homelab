# Day 10 — Phase 2: SIEM Architecture Pivot & Exit Criteria Close

**Date:** July 15, 2026

## Goals for today

- Execute yesterday's decision: SIEM01's Wazuh install failed on a defective installer, and separately, Wazuh's actual 8GB RAM requirement had exceeded the 4GB originally budgeted for this host — an architecture pivot to a self-hosted Loki + Grafana + Alloy stack was decided rather than continuing to fight the Wazuh install
- Close Phase 2's real exit criteria (SIEM ingesting from both DCs, confirmed) today, tomorrow at the absolute latest — this has already displaced Phase 4/5 timing once
- Treat the tooling swap as complete only once every document that referenced Wazuh actually agrees with the lab, not the moment ingestion is confirmed

## What got done

1. Patched SIEM01 (246 pending updates) before making any further changes, then revisited RAM sizing left over from the pivot decision. Weighed 2GB against 4GB by checking real container usage rather than guessing from spec sheets, and landed on 4GB. This happened to restore the exact figure Doc 04 originally budgeted for this host back in Phase 1 — Wazuh's 8GB requirement had quietly exceeded that budget without ever being reconciled in the doc.

2. Deployed Loki + Grafana via Docker Compose and added Loki as a Grafana data source (`http://loki:3100`, container-network hostname). Hit a YAML parsing defect in `docker-compose.yml` twice — inconsistent indentation under a sibling key — before rewriting the file cleanly via a heredoc rather than continuing to hand-edit it in nano.

3. Mid-patch, SIEM01 rebooted and Grafana/Loki didn't come back up on their own. Diagnosed via `uptime` and `docker ps -a`: no restart policy was defined. Added `restart: unless-stopped` to both services — validated for real at end of day, when SIEM01 was rebooted a second time and both containers came back up unassisted.

4. Reviewed pfSense rules across LAN/OPT1/OPT2 before touching Alloy. Confirmed no firewall change was needed for DC01-IRONBRIDGE (same L2 segment as SIEM01). Added a new rule scoped to the single HAWTHORNE host — `10.50.30.10 → 10.50.10.20:3100 TCP` on OPT2 — for its cross-VLAN path. Disabled, rather than deleted, the legacy Wazuh agent ports (1514/1515) on OPT1 and OPT2, preserving the audit trail instead of erasing what was originally provisioned.

5. Neither DC has general outbound internet access by design, so the Alloy installer had to be transferred in rather than downloaded directly. Built a small ISO with `genisoimage` and attached it as a virtual CD-ROM via virt-manager/virsh — the recurring `qemu:///session` vs. `qemu:///system` distinction was still relevant here.

6. Installed Grafana Alloy on both DC01-IRONBRIDGE and DC01-HAWTHORNE, using an identical `config.alloy` on both — `loki.source.windowsevent` across Security/System/Application channels, pushed to `http://10.50.10.20:3100/loki/api/v1/push`, labeled with `instance = constants.hostname` so each host is independently distinguishable in Loki rather than blended together.

7. Caught and corrected a typo on HAWTHORNE's config (`use_incoming>timestamp` instead of `use_incoming_timestamp`) before it caused a silent parse failure on that host.

8. Started the Alloy service on both DCs and confirmed in Grafana Explore that real Windows Event Log entries were landing from both, correctly tagged by `instance`. This is the actual Phase 2 exit criterion — met and confirmed in the browser, not assumed.

9. Ran SIEM01, FW01, and both DCs concurrently for the first time in this build; virt-manager crashed mid-session and had to be reconnected. Noted as a genuine constraint to plan around for Phase 3 — which VMs actually need to run together per session — rather than something to just push through as-is.

10. Built one Grafana dashboard panel ("IHA Phase 2 - Windows Event Log Overview") — Logs visualization, timestamps on, querying both DCs together. Satisfies the dashboard-verification exit criterion without over-building toward Phase 3's alerting work.

11. Committed `configs/siem01/docker-compose.yml` and `configs/alloy/config.alloy` to GitHub via the web editor — a deliberate call to skip local git setup for now rather than add friction to an already full day.

12. Shut down SIEM01 cleanly and took a snapshot (`siem01-phase2-exit-baseline`), description covering the full pivot rationale, then restarted it as the final proof of the restart-policy fix under a real power cycle.

Spent the evening reconciling every document that still referenced Wazuh, rather than calling the pivot finished once ingestion was confirmed: Doc 04 (system build specs), Doc 05 (GRC mapping — re-scoped the audit logging control rather than just renaming it, since the new stack doesn't provide alerting or FIM the way Wazuh did), Doc 06 (added risk register IHA-011/IHA-012 and threat model T-7), Doc 07 (added `configs/` and `evidence/` to the folder structure), the Network Addressing & Firewall Rule Set doc (disabled-rule status, the new 3100 rule, a pivot design note), the topology overview SVG (SOC/NOC label corrected so it doesn't imply alerting that doesn't exist yet), and the README folder-contents table.

## Notes

Day 8's notes flagged a pattern worth watching: two sessions in a row where one small task uncovered a deeper issue that ate the whole day, with Phase 2's actual exit criterion not moving either time. Today carried a structurally bigger disruption than either of those — a full SIEM architecture pivot, not just an unplanned investigation — and the exit criterion still closed, on the day set as the hard deadline. Worth naming plainly since it's evidence the earlier lesson actually held: protect the real exit criterion, and let everything else be scoped around it rather than the reverse.

Equally worth naming: today's harder discipline wasn't the technical pivot itself, it was not stopping at "ingestion confirmed." Wazuh's name was written into six different places — two GitHub docs, two Excel workbooks, a topology diagram, and a README table — and each one made a slightly different implicit claim about what the environment could do. Chasing all six down, rather than treating the working dashboard as proof the job was done, is the same instinct the GRC and risk register work depends on: the environment and the paperwork describing it have to actually agree, or the paperwork is worth nothing the moment someone checks.
